# Break Down a Problem

Big problems are intimidating. When you stare at a project description and think
"I have no idea where to start," that's completely normal. Professional
developers feel the same way — they've just learned to break big problems into
small, manageable pieces.

This skill is about taking something that feels overwhelming and turning it into
a checklist of things you *can* do, one at a time.

## When to Use This Skill

- A project or assignment feels too big to start
- You've been staring at your screen for ten minutes without typing anything
- You're not sure what order to do things in
- You have a working version but don't know how to add the next feature

## Step-by-Step Process

### 1. Identify the End Goal Clearly

Before you start breaking things apart, make sure you know what "done" looks
like. Write it down in one sentence: "When this is finished, it will ___."

If you can't finish that sentence, you need more clarity before you start
coding. Ask your teacher, re-read the assignment, or talk it through with a
classmate.

### 2. List What You Know and Don't Know

Split a page (or a doc) into two columns:
- **What I know**: tools I'll use, concepts I understand, similar things I've
  built before
- **What I don't know**: things I need to look up, parts I've never tried,
  questions I have

This is honest and useful. The "don't know" list becomes your research tasks.

### 3. Split into Smaller Tasks

Take your big goal and break it into pieces. Each piece should be something you
could describe in one sentence and finish in a short work session. If a task
still feels too big, break it down further.

A good task sounds like:
- "Create the HTML structure for the calculator"
- "Style the buttons with CSS"
- "Write a function that adds two numbers"

A bad task sounds like:
- "Build the calculator" (too big)
- "Do the code" (too vague)

### 4. Put Tasks in Order (What Depends on What?)

Some tasks need to happen before others. You can't style buttons that don't
exist yet. You can't test a function you haven't written. Line up your tasks
so each one builds on the last.

Look for tasks that are independent — those can happen in any order, which gives
you flexibility.

### 5. Start with the Smallest Task

Pick the easiest task on your list and do it. Getting that first small win builds
momentum. You'll feel less stuck after finishing even one piece.

Don't start with the hardest part. Warm up with something manageable, then work
your way into the tougher stuff.

### 6. Check Progress and Adjust

After finishing a few tasks, pause and look at your list. Do you need to add
new tasks you didn't think of? Remove ones that turned out to be unnecessary?
Re-order things? Your plan is a living document — update it as you learn more.

## Example Scenario

Your assignment: build a calculator app using HTML, CSS, and JavaScript. That
feels like a lot. Here's how to break it down:

**End goal**: A webpage with buttons for digits and operations that displays
the result of basic math.

**Tasks**:
1. Create the HTML layout — a display area and a grid of buttons (0-9, +, -, *,
   /, =, clear)
2. Style the calculator with CSS — make it look like an actual calculator with
   a clean grid layout
3. Write JavaScript to capture button clicks and show the digit in the display
4. Handle the math operations — store the first number, the operator, and
   compute when = is pressed
5. Add the clear button — reset the display and stored values
6. Test edge cases — what if someone presses = with no numbers? What about
   dividing by zero?

Each of these is a focused task you can work on and finish. Task 1 takes maybe
15 minutes. Task 4 might take 30. None of them require you to think about the
whole project at once.

## Tips for Effective Problem Breakdown

- **Write your tasks down.** In a notebook, a text file, or sticky notes — just
  get them out of your head and somewhere visible.
- **Each task should be completable in under 30 minutes.** If it feels longer,
  split it again. Smaller is almost always better.
- **It's okay to re-split.** You'll often discover mid-task that it's bigger than
  you thought. That's not failure — that's learning. Break it down further and
  keep going.
- **Celebrate small wins.** Checking off a task feels good. Let it. That momentum
  carries you to the next one.
- **Talk it out.** If you're stuck on how to break something down, explain the
  project to someone else. Teaching forces clarity.

No crew ever built a ship all at once. Plank by plank, mate — that's how it's
done.
